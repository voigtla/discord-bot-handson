# 全10回ロードマップ

## コンセプト

> **メンタル系サーバー向けチャットボットを、「壊れず・覚えて・外とつながる」形に育てていく10回**

- 毎回「1つだけ」やる
- DB・API・インフラは **必要になった瞬間にだけ出す**
- 理論より **「そうしないと詰む」を体験ベースで**

---

## 前提条件

### 参加者の状況
- 毎回 **15名程度・固定ではない**
- 途中参加・途中離脱が前提
- Discord Bot実装経験あり
- JSONでのデータ管理に限界を感じている
- Gemini API / 外部API への関心が高い

### これからやりたいこと
- Discord Bot × DB
- Discord Bot × LLM（Gemini）
- 最終的に GCP（SSH含む）を視野

### 制約条件
- **GCP環境はまだ準備できていない**
- → 後半は「行ける人向けの地図」を出す設計

---

## 各回の詳細

### 第1回・第2回について

**第1回**: 基礎的すぎるため事実上スキップ

**第2回**: Git/GitHub基礎（今回）
- 実質のスタート地点
- チーム開発の土台を作る

---

### 第3回：Bot導入＋SQLiteで「1つだけ覚える」

⭐ **実質スタート回**

#### やること
- `/hello` コマンドを再確認
- SQLite を導入
- **「1回だけ」DBに保存する処理**

**例:**
```javascript
// ユーザーが /hello を打つ
// → ユーザーIDと時刻をDBに保存
// → 返事は固定文でOK
```

#### やらないこと
- DB設計論
- 複雑なSQL
- 複数テーブル

#### 👉 ゴール
**「Botが初めて"記憶"を持つ」**

---

### 第4回：メンタル系Botとしての最小実装

⭐ **テーマ回**

#### やること
- メンタル系サーバー前提の設計
- 超シンプルな機能を1つ

**例:**
```javascript
// /checkin → 今日の気分を数字 or 選択肢で保存
// /last → 前回の状態を返す
```

#### ポイント
- 判断しない
- 評価しない
- ただ保存して返す

#### 👉 ゴール
**「安全なチャットボットの型」を知る**

---

### 第5回：DBを使った"意味のある返答"

#### やること
- SQLiteの読み取り
- ユーザーごとの状態を返す

**例:**
```javascript
// 「前に来たことある人」には別の返事
// 回数カウント
// 最終利用日時の表示
```

#### 👉 ゴール
**「DBがあると何が嬉しいかを実感」**

---

### 第6回：Gemini API で「優しい返答」を自動生成

⭐ **LLM導入回**

#### やること
- Gemini API を叩く
- DBの内容を渡して返答を生成

**例:**
```javascript
// 最近のチェックイン内容を要約
// 優しい言い換え文を生成
```

#### やらないこと
- プロンプト職人芸
- 精度議論

#### 👉 ゴール
**「LLMは魔法じゃなく関数」**

---

### 第7回：エラー処理・リトライ・ログ保存の実装

#### やること
- API呼び出しのエラーハンドリング
- リトライロジック
- ログの保存方法

**例:**
```javascript
// Gemini APIがタイムアウトしたら？
// レート制限に引っかかったら？
// エラーをどう記録する？
```

#### 👉 ゴール
**「なんとなく動いてる」から脱出**

---

### 第8回：安全・倫理・事故らない設計

⭐ **メンタル系サーバー特化回**

#### やること
- NGな実装例
- 「やらない」判断の基準
- ログ・保存・可視化の扱い

**例:**
```
⭕ やっていいこと: 要約、言い換え、共感表現生成
❌ やってはいけないこと: 診断、判断、アドバイス
⚠️ 慎重に扱うこと: 履歴の要約、パターン検出
```

#### 👉 ゴール
**「作れるけど、やらない」を選べる**

---

### 第9回：このBotを"外に置く"としたら

⭐ **インフラ前夜**

#### やること
- ローカルBotの限界
- GCP / VPS / Railway などの選択肢
- SSHって何をするものか

**注意:**
- 実作業はしない
- **地図だけ渡す**

#### 👉 ゴール
**「次に何が必要か分かる」**

---

### 第10回：振り返り＋分岐点を渡す

#### やること
- JSON時代 → DB → LLM の変遷を振り返る
- 各自の現在地確認
- 次の分岐提示

**分岐例:**
- 本番運用
- GCP実装
- 別API連携
- Botを作らない判断

#### 👉 ゴール
**「自分で続ける or やめるを選べる」**

---

## この設計の強み

### 1. ランダム参加でも壊れない
- 各回が独立しつつ、連続性も保つ
- 途中参加者向けの導線がある

### 2. DBとLLMを「必要だから」使う流れ
- 理論→実践ではなく、課題→解決の順

### 3. GCP未準備でも詰まらない
- 第9回で「地図だけ渡す」設計
- 無理に環境を用意して失敗するより健全

### 4. メンタル系という制約が技術と倫理を結びつける
- 「作れる」と「やっていい」の違いを自然に学べる
