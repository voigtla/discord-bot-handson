# 第20回：自分がいなくなっても Bot が回る状態を作る

―― 開発の最後の仕事 ――

Bot は動いている。
運用もできる。
判断もできる。

でも最後に、必ず来る問いがあります。

> **「もし自分が消えたら、このBotどうなる？」**

この回では、

* 新しい機能は作らない
* コードはほとんど触らない

👉 **「未来の他人」に仕事を渡す準備**をします。

---

## 📌 この回の目標

* Bot を **他人が引き継げる状態**にする
* 「最低限あれば止まらない」ドキュメントを作る
* **自分が抜けても罪悪感が残らない**状態を作る

**💡 この回の約束**

* 綺麗な資料は作らない
* 全部説明しない
* README を完璧にしない

👉 **“止まらない最低限”だけ**を作ります。

---

## 🎯 完成イメージ

```
【Before】
自分しか分からない
触るのが怖い
止まったら詰む

【After】
README がある
起動方法が書いてある
困ったらここを見る
```

---

## 第1章：「全部分かる人」はいなくなる（10分）

### 1-1. 現実の未来

* 忙しくなる
* 興味が移る
* チームを離れる

これは **失敗ではない**。

---

### 1-2. ドキュメントの役割

ドキュメントは：

* 知識共有 ❌
* 教育資料 ❌

ではなく、

> **「止まった時の取扱説明書」**

---

## 第2章：未来の他人を決める（5分）

この回で想定する人物：

> **「このBotを全く知らないが、
> SSH はできる人」**

* Node.js は詳しくない
* DB設計も知らない
* でもサーバーには入れる

👉 **この人が基準**

---

## 第3章：README を“最低限”書く（20分）

### 3-1. README に書くのはこれだけ

```markdown
# このBotについて

## 何をするBotか
（1〜2行）

## 起動方法
- サーバーに入る
- このディレクトリ
- このコマンド

## 停止方法
- このコマンド

## よくあるトラブル
- 起動しないとき
- 反応しないとき
```

---

### 3-2. 例（雑でOK）

```markdown
## 起動方法
pm2 start ecosystem.config.js

## 停止方法
pm2 stop bot-name

## 反応しないとき
pm2 logs bot-name を見る
```

👉 **説明しすぎない**

---

## 第4章：運用メモを1枚作る（15分）

### 4-1. runbook.md を作る

```markdown
# 運用メモ

## 本番サーバー
- GCP VM
- pm2 使用

## DB
- SQLite
- data/bot.db

## デプロイ方法
- deploy.sh を実行

## バックアップ
- 手動コピー
```

---

### 4-2. 書けなかったら、それが重要

* 書けない
* 思い出せない

👉 **それは危険ポイント**

---

## 第5章：判断ログをまとめる（15分）

### 5-1. decision.md を作る

```markdown
# 判断ログ

## DB設計
- 正規化しない
- SQLite 継続

## 運用方針
- 1インスタンス複数Bot可
- 落ちたらまず止める

## 継続判断
- 2026-xx-xx 継続
```

👉 **正解である必要はない**

---

## 第6章：本当に不要なものを消す（10分）

### 6-1. 消していいもの

* 使っていないコメント
* 放置された TODO
* 未使用ファイル

👉 **混乱の元を減らす**

## 第7章：この回で一番大事な問い（10分）

* 自分が消えても **罪悪感ない？**

> 手放せる状態**
